Value Reference Diagnostics Plan

Unresolved Dagu value references should stay non-blocking, but they should not look like noisy errors or warnings. This plan replaces warning-only misses with passive diagnostics that are visible only when users choose to inspect them.

Plan version: 2 Status: approved by stuff-engineer review Target: low-noise diagnostics

Decision

Dagu should preserve unresolved supported value references and record a low-severity diagnostic only when the owning field is actually evaluated and the value is still unavailable.

Validation must not report a diagnostic just because a value may exist later at runtime. Static validation should reject invalid declarations and invalid shapes, not runtime-dependent values.

Error

Blocks validation or execution. Use for invalid YAML, invalid field shape, invalid declarations, executor failure, output parse failure, and typed-field parse failure.

Warning

Non-blocking but likely wrong. Use for deprecated syntax or configuration that is valid today but should be changed.

Notice

Non-blocking diagnostic. Use for unresolved supported value references after Dagu evaluated the field and preserved the original token.

Behavior Contract

Situation Behavior User-visible level
Unsupported syntax such as $params.name or ${params...} Dagu preserves the text as ordinary content. None
Supported reference may resolve later, such as ${params.name} during static validation Dagu preserves the text for later evaluation. None
Supported reference is unresolved when the owning field is evaluated Dagu preserves the original token and records a diagnostic. Notice
Deprecated but still accepted syntax Dagu accepts the workflow and records migration guidance. Warning
Invalid declaration, invalid shape, invalid YAML, failed executor, or failed output parsing Dagu rejects validation or marks the relevant execution failed. Error

Diagnostic Shape

The diagnostic record should be structured so the Web UI can filter and group it without parsing strings.

type Diagnostic struct {
    Level   DiagnosticLevel // error, warning, notice
    Code    string          // value_reference_unresolved
    Field   string          // steps.build.run
    Token   string          // ${params.image_tag}
    Message string
}

Use value_reference_unresolved for unresolved supported value references. The code describes the user-visible condition. It does not describe the internal action Dagu took.

Example message: ${params.image_tag} was left unchanged because params.image_tag had no value when steps.build.run was evaluated.

The deduplication key is level + code + field + token. Deduplication happens within one field evaluation and again when persisted run diagnostics are appended.

Phase Rules

Phase What Dagu should diagnose What Dagu should not diagnose
Validation Invalid declarations, invalid field shapes, and deprecated syntax. Runtime params, runtime env, and step outputs that may exist later.
Start Fields evaluated before the run starts when a supported reference still has no value. Step output references whose producing step has not run yet.
Step runtime Step-owned fields when Dagu evaluates that step and a supported reference still has no value. Unsupported language syntax and shell-owned variables.
Output collection Output fields that Dagu evaluates after a step finishes and a supported reference still has no value. Normal absence of an optional output unless a field explicitly references it.

Namespace Matrix

This matrix prevents static validation from becoming noisy while keeping hard contract errors intact.

Namespace Validation behavior Evaluation behavior
consts Invalid const declaration shape is an error. Unknown or later const references are not hard validation errors. If the const value is unavailable when the field is evaluated, preserve the token and record a notice.
params Invalid param declaration name or shape is an error. A missing runtime param value is silent during validation. If the param value is unavailable when a start-time or runtime field is evaluated, preserve the token and record a notice.
env Invalid env field shape is an error. A missing process or DAG env value is silent during validation. If the env value is unavailable when the owning field is evaluated, preserve the token and record a notice.
steps.*.outputs Invalid output declaration shape is an error. Unknown step, undeclared output, self-reference, and non-upstream reference do not block validation by themselves. If the output is unavailable when the owning field is evaluated, preserve the token and record a notice. Output parse and output schema failures remain errors.
Unsupported syntax Dagu treats it as ordinary content. Dagu preserves it silently.

Backend Design

  1. Introduce a shared diagnostic model with level, code, field, token, and message.
  2. Keep existing hard errors as errors. Do not downgrade invalid specs, executor failures, or output parse failures.
  3. Move unresolved supported value-reference misses from warning strings into notice diagnostics.
  4. Keep static validation quiet for references that may resolve later. Runtime field evaluation should diagnose only references that are still unavailable.
  5. Add a resolver diagnostic sink. Runtime callers pass the sink when evaluating a field.
  6. Thread that sink through runtime env, shell, command, condition, executor config, handler, working directory, and output collection helpers.
  7. Deduplicate diagnostics by level, code, field, and token within one field evaluation.
  8. Keep logger output for warnings and errors. Do not log notices by default unless debug logging is enabled or an explicit diagnostic sink requests them.
type DiagnosticSink interface {
    AddDiagnostic(Diagnostic)
}

type ResolveOptions struct {
    Field       string
    Diagnostics DiagnosticSink
}

Existing resolver entry points may keep their value-and-error return shape, but runtime evaluation must pass ResolveOptions. A resolver miss adds a notice to the sink and returns the preserved value.

Persistence Design

Runtime diagnostics must survive after execution. They also need to survive distributed execution.

  1. Add diagnostics to the persisted DAG-run status or an equivalent run metadata object.
  2. Append diagnostics during start-time evaluation, step runtime evaluation, handler execution, and output collection.
  3. Carry diagnostics through local runner status writes.
  4. Carry diagnostics through worker-to-coordinator reporting.
  5. Expose diagnostics for sub-DAG runs without merging them into the parent run's errors.
  6. Include diagnostics in SSE updates and ToDAGRunDetails.
  7. Deduplicate persisted diagnostics by level, code, field, and token.

API Design

API responses must not mix notices or warnings into errors. The Web UI needs structured diagnostics so visual severity follows the data model.

Keep existing errors: string[] fields for compatibility. They contain only blocking errors. Add diagnostics: Diagnostic[] for warnings and notices. Do not require old clients to parse diagnostics.

{
  "errors": [],
  "diagnostics": [
    {
      "level": "notice",
      "code": "value_reference_unresolved",
      "field": "steps.build.run",
      "token": "${params.image_tag}",
      "message": "${params.image_tag} was left unchanged because params.image_tag had no value when steps.build.run was evaluated."
    }
  ]
}

DAG details, DAG spec, DAG validation, start responses, and DAG run details should use this shape where they expose diagnostics. Existing build warnings become warning-level diagnostics. Value-reference misses become notice-level diagnostics.

CLI Design

  • dagu validate prints blocking errors and warning-level diagnostics. It does not print notice-level unresolved reference diagnostics for runtime-dependent values.
  • dagu start records notice-level runtime diagnostics, but does not print them as normal stderr warnings by default.
  • dagu dry follows validation behavior unless it evaluates a field with concrete runtime data.
  • dagu status may expose diagnostics through an explicit verbose, JSON, or diagnostics view.
  • Debug logging may include notice diagnostics. Default logs should not make notices look like errors or warnings.

Web UI Design

Diagnostics should be available when users want them. They should not dominate the page or make a valid DAG look broken.

  • Do not show notices as red banners.
  • Do not block graph rendering because notices exist.
  • Do not disable Start because notices exist.
  • Show errors inline where the user must act.
  • Show warnings as amber, non-blocking guidance.
  • Hide notices by default behind a small Diagnostics affordance with a count.
  • Let users open Diagnostics and filter by errors, warnings, and notices.
  • Group value-reference notices by field.

Typed Field Rule

Preserving an unresolved token does not make an invalid typed value valid.

If a typed field needs an integer, boolean, duration, enum, path, output schema, or other validated type, the typed consumer may still fail after value resolution preserves the token.

That later failure is an error. The notice only explains why the original reference text was still present.

Implementation Tasks

Task 1: Add the diagnostic domain model Foundation

Create a typed diagnostic representation and helper constructors for errors, warnings, and notices. Keep message formatting centralized enough that API and CLI output stay consistent.

  • Acceptance: unresolved value-reference diagnostics can carry level, code, field, token, and message.
  • Acceptance: existing build warnings can still be represented without losing their severity.
  • Verification: unit tests cover construction, deduplication keys, and JSON shape.

Task 2: Make value-resolution diagnostics passive Core

Change unresolved supported reference handling so validation does not diagnose references that may resolve later. Runtime field evaluation records a notice through a diagnostic sink when the value is still unavailable.

  • Acceptance: ${params.name} during static validation is silent when it may resolve at start.
  • Acceptance: missing ${params.name} during actual field evaluation records a notice and preserves the token.
  • Acceptance: unsupported syntax remains silent.
  • Acceptance: resolver callers in env, shell, command, condition, handler, working directory, executor config, and output collection can pass a diagnostic sink.
  • Verification: resolver and core validation tests cover static validation, start-time evaluation, and step runtime evaluation.

Task 3: Persist runtime diagnostics Runtime

Store runtime diagnostics with DAG-run state so notices remain inspectable after the run and across distributed execution.

  • Acceptance: local runner status writes include diagnostics.
  • Acceptance: worker and coordinator reporting preserves diagnostics.
  • Acceptance: sub-DAG diagnostics remain associated with the sub-DAG run.
  • Verification: runtime and distributed tests assert diagnostics survive status reload.

Task 4: Separate errors and diagnostics in API responses API

Stop appending build warnings or notices into errors. Add structured diagnostics to the relevant OpenAPI schemas and regenerate API types.

  • Acceptance: legacy errors: string[] fields contain only blocking errors.
  • Acceptance: DAG spec, DAG details, start responses, and DAG run details expose diagnostics: Diagnostic[].
  • Acceptance: generated frontend types include the diagnostic shape.
  • Verification: API handler tests assert warnings and notices do not appear in errors.

Task 5: Add passive Web UI diagnostics UI

Add a diagnostics affordance on DAG and DAG-run pages. It should be visible enough to discover, but collapsed by default and visually separate from errors.

  • Acceptance: notices do not render as destructive alerts.
  • Acceptance: notices do not block graph rendering or starting a DAG.
  • Acceptance: users can open Diagnostics and view grouped notice details.
  • Verification: component tests cover errors, warnings, notices, and empty diagnostics.

Risks

Risk Impact Mitigation
Diagnostics are too hidden. Users may miss useful troubleshooting data. Show a small Diagnostics count in DAG and run pages. Keep details collapsed by default.
Diagnostics are too noisy. Users learn to ignore them. Do not emit notices during validation for values that may resolve later. Keep unsupported syntax silent.
API compatibility changes affect frontend callers. Existing code may expect warnings inside errors. Introduce separate fields while keeping errors strictly for blocking errors. Update generated types and callers together.
Runtime notices are lost after execution. Users cannot inspect what happened after the run ends. Persist run diagnostics with DAG-run state or an existing run metadata path before exposing them in run details.
