Value Resolution Params Conformance Plan

This plan describes the product-level work needed for ${params.name} references to conform to Spec 005 and the shared value-resolution rules in Spec 003. It intentionally avoids implementation design and file-level migration instructions.

Invariant: Spec 005 wins over current gaps Current status: target behavior not implemented Plan level: observable behavior only

End-to-End Success Condition

Workflow authors can use ${params.name} in every supported value-resolution field after declaring the named runtime parameter. Dagu rejects invalid or undeclared parameter references during validation, fails missing runtime values before the owning field is used, and keeps existing bare-name parameter compatibility separate from the new params namespace.

Contract Sources

Source Plan obligation
Spec 003: Value Resolution Use the shared Dagu-owned reference syntax, supported-field list, resolution timing, validation expectations, and string insertion rules.
Spec 005: Value Resolution Params Define ${params.name} as a named runtime parameter lookup with declaration validation, runtime availability, and missing-value errors.
Spec Implementation Guide When current behavior conflicts with a numbered spec, the spec wins. Conformance claims require black-box coverage, and plans should not expose implementation internals as public behavior.

Current Product Facts

Runtime parameters already exist. Dagu already accepts runtime parameters, supports named declarations in current workflows, validates runtime input in several entry points, and rejects unknown named runtime input when named declarations are present.
Legacy bare-name behavior exists. Current behavior can expose parameter values through existing bare-name or environment-style expansion surfaces. That compatibility behavior is not the same contract as ${params.name}.
The namespaced params contract is still target behavior. The current product must not be described as conforming to Spec 005 until declared ${params.name} references resolve and fail according to the spec.
Some current defaults are compatibility-only. Empty-string seeding for omitted named parameters and numeric handling for positional parameters must not be applied to ${params.name}.

Target Behavior

Area Required outcome
Reference form ${params.name} is a Dagu-owned reference. name must be a declared named parameter and must match ^[A-Za-z][A-Za-z0-9_]*$.
Invalid shorthand $params.name and other supported-namespace shorthand forms are invalid Dagu-looking syntax and fail workflow validation. Ordinary environment syntax such as $NAME is a separate behavior.
Declaration boundary Only named parameter declarations are addressable. Positional parameters are not reachable through the params namespace.
Declaration values Parameter declarations do not support Dagu references. Declaration validation is separate from runtime value availability.
Supported fields ${params.name} resolves in every Spec 003 value-resolution field whose timing occurs after runtime params are available. Root consts list values are covered as a negative case because they can see only earlier consts entries. Fields outside the Spec 003 list remain unsupported unless the spec changes.
Resolution timing Runtime parameter values become available after Dagu builds the run input. Supported fields resolve only when the field is about to be used, not by rendering the whole workflow file in advance.
Validation Invalid declaration names, undeclared names, invalid shorthand, unavailable-field references, and named references in positional-only workflows fail validation when statically knowable. Braced text that is not a Dagu-owned reference remains ordinary string content.
Missing values A declared ${params.name} reference with no runtime value fails before the owning field is used.
String insertion Strings, booleans, integers, and non-integer numbers are inserted into string fields using the shared Spec 003 rules.
Environment syntax Single-quoted $NAME and ${NAME} environment references remain preserved during Dagu environment expansion.

Compatibility Boundaries

  • Bare names, $NAME, and unqualified ${NAME} continue to belong to their existing environment, shell, or legacy parameter behavior. They are not aliases for ${params.name}.
  • $params.name is not ordinary environment syntax. It is invalid Dagu-looking shorthand and must fail validation in value-resolution fields.
  • Root consts list values do not resolve ${params.name}; they can reference only earlier consts entries.
  • Current bare-name parameter compatibility can be preserved only where it does not conflict with the numbered specs.
  • Template defaults, empty values for omitted named parameters, and positional numeric names do not satisfy a missing ${params.name} reference.
  • Expression syntax outside supported Dagu-owned reference forms remains out of scope unless a numbered spec adds it.
  • The validator and runtime must agree on the same supported-field list so a reference accepted by validation is not silently ignored later.

Acceptance Scenarios

Scenario Expected observable result
Declared named parameter in a step command A workflow with environment declared and ${params.environment} in a supported step field uses the runtime value supplied for environment.
Declared named parameter in supported non-command fields Root env, step env, dotenv paths, working directories, preconditions, container fields, parallel parameter values, sub-workflow parameters, and output-related strings resolve consistently with Spec 003 timing.
Undeclared parameter reference dagu validate fails for ${params.region} when only environment is declared.
Invalid parameter declaration name Validation fails for names that do not match the Spec 005 identifier rule.
Invalid params shorthand Validation fails for $params.environment in a value-resolution field, while ordinary environment syntax remains governed by the field's existing environment or shell behavior.
Params reference in consts Validation fails for ${params.environment} in a root consts list value because params are unavailable while consts are resolved.
Positional parameter through params namespace Validation fails because positional runtime parameters have no named declaration in the params namespace.
Missing runtime value A declared ${params.name} reference fails before the owning field is used when the run input does not provide a value.
Legacy bare-name parameter compatibility Existing bare-name behavior remains unchanged where it is not contradicted by the spec, and it does not mask missing ${params.name} values.
String insertion String, boolean, integer, and non-integer numeric runtime values appear in string fields using the Spec 003 representation.

Work Phases

  1. Baseline current behavior: document black-box facts for existing runtime parameter input, bare-name compatibility, validation, and supported fields without treating current gaps as target behavior.
  2. Define validation conformance: make validation accept valid declared ${params.name} references and reject invalid declarations, undeclared names, unavailable-field references, shorthand, or named references in positional-only workflows.
  3. Define runtime conformance: make supported fields resolve declared runtime parameter values at field-use time and fail missing values before the field is used.
  4. Protect compatibility boundaries: preserve existing bare-name parameter behavior where allowed, while proving it does not alter the ${params.name} contract.
  5. Close conformance: mark Spec 005 as implemented only after the behavior above has black-box coverage across validation and runtime paths.

Black-Box Test Plan

  • Add runnable workflow fixtures for every acceptance scenario above and for the full Spec 003 supported-field list, including the negative root consts case.
  • Cover every root, step, handler, environment, container, parallel, sub-workflow, and output-related field where Spec 005 makes params available.
  • For every field where params are available, cover valid declared-name resolution, undeclared-name validation, invalid shorthand validation, and missing-runtime-value failure before field use.
  • Name and justify any hard-to-test exception, keep it small, and do not count it as completed conformance until the exception is resolved or accepted.
  • Assert validation failures by failure class and field path, not by implementation-specific error text.
  • Assert legacy bare-name behavior separately from ${params.name} behavior so compatibility does not obscure conformance.
  • Keep examples aligned with Spec 005 so they can become conformance fixtures.

Risk Register

Risk Mitigation
Current unresolved ${params.*} behavior is accidentally preserved as the target. Keep Spec 005 as the source of truth and require tests that prove declared names resolve.
Bare-name compatibility changes the new namespace semantics. Test bare names and ${params.name} separately, especially omitted values and positional parameters.
Validation and runtime support different field lists. Use field-list conformance tests that exercise validation and execution for representative supported fields.
Missing runtime values fail too late. Acceptance tests must prove the owning field is not used after a missing ${params.name} value is detected.
Spec status is overstated. Keep Spec 005 marked not implemented or partial until every required behavior has black-box coverage.
