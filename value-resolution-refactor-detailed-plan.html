Spec 005 Value Resolution Params Implementation Plan

This plan sequences the product work needed for ${params.name} references to conform to Spec 005 and the shared value-resolution rules in Spec 003. It stays at the observable behavior and conformance level: it names required outcomes, ordering, verification, and compatibility boundaries without exposing private implementation design as public contract.

Invariant: Spec 005 wins over current gaps Current status: target behavior not implemented Plan level: behavior and conformance

End-to-End Success Condition

Workflow authors can use ${params.name} in every supported value-resolution field after declaring the named runtime parameter. Dagu rejects invalid or undeclared parameter references during validation, fails missing runtime values before the owning field is used, and keeps existing bare-name parameter compatibility separate from the new params namespace. Unbraced namespace-looking text such as $params.name remains ordinary content because Dagu owns only supported braced reference forms.

Contract Sources

Source Plan obligation
Spec 003: Value Resolution Use the shared Dagu-owned reference syntax, supported-field list, resolution timing, validation expectations, and string insertion rules.
Spec 005: Value Resolution Params Define ${params.name} as a named runtime parameter lookup with declaration validation, runtime availability, and missing-value errors.
Spec Implementation Guide When current behavior conflicts with a numbered spec, the spec wins. Conformance claims require black-box coverage, and plans should not expose implementation internals as public behavior.

Current Product Facts

Runtime parameters already exist. Dagu already accepts runtime parameters, supports named declarations in current workflows, validates runtime input in several entry points, and rejects unknown named runtime input when named declarations are present.
Legacy bare-name behavior exists. Current behavior can expose parameter values through existing bare-name or environment-style expansion surfaces. That compatibility behavior is not the same contract as ${params.name}.
The namespaced params contract is still target behavior. The current product must not be described as conforming to Spec 005 until declared ${params.name} references resolve and fail according to the spec.
Some current defaults are compatibility-only. Empty-string seeding for omitted named parameters and numeric handling for positional parameters must not be applied to ${params.name}.

Target Behavior

Area Required outcome
Reference form ${params.name} is a Dagu-owned reference. name must be a declared named parameter and must match ^[A-Za-z][A-Za-z0-9_]*$.
Unbraced namespace-looking text $params.name, $consts.name, $env.NAME, and $steps.step.outputs.name are ordinary string content, not Dagu-owned reference syntax. Dagu may report a non-fatal warning for this text when a warning channel exists, but validation and execution must not fail because of it alone.
Declaration boundary Only named parameter declarations are addressable. Positional parameters are not reachable through the params namespace.
Declaration values params[].default and declaration metadata do not support Dagu references. params[].eval follows the dynamic-evaluation specs, and Dagu references inside it use the Spec 005 declaration and runtime lookup contract.
Supported fields ${params.name} resolves in every Spec 003 value-resolution field whose timing occurs after runtime params are available. Root consts list values are covered as a negative case because they can see only earlier consts entries. Fields outside the Spec 003 list remain unsupported unless the spec changes.
Resolution timing Runtime parameter values become available after Dagu builds the run input. Supported fields resolve only when the field is about to be used, not by rendering the whole workflow file in advance.
Validation Invalid declaration names, undeclared braced names, unavailable-field references, and named references in positional-only workflows fail validation when statically knowable. Braced text that is not a Dagu-owned reference and unbraced namespace-looking text remain ordinary string content.
Missing values A declared ${params.name} reference with no runtime value fails before the owning field is used.
String insertion Strings, booleans, integers, and non-integer numbers are inserted into string fields using the shared Spec 003 rules.
Environment syntax Single-quoted $NAME and ${NAME} environment references remain preserved during Dagu environment expansion.

Compatibility Boundaries

  • Bare names, $NAME, and unqualified ${NAME} continue to belong to their existing environment, shell, or legacy parameter behavior. They are not aliases for ${params.name}.
  • $params.name is not a Dagu reference. It remains ordinary string content so workflow scripts in languages such as PHP or Perl can use their own variable syntax without Dagu rejecting the workflow.
  • Root consts list values do not resolve ${params.name}; they can reference only earlier consts entries.
  • Current bare-name parameter compatibility can be preserved only where it does not conflict with the numbered specs.
  • Template defaults, empty values for omitted named parameters, and positional numeric names do not satisfy a missing ${params.name} reference.
  • Expression syntax outside supported Dagu-owned reference forms remains out of scope unless a numbered spec adds it.
  • The validator and runtime must agree on the same supported-field list so a reference accepted by validation is not silently ignored later.

Implementation Sequence

Order Workstream Observable result Conformance gate
1 Keep the conformance suite red and authoritative. Running make test-conformance executes the binary-level Spec 005 tests with no skips. Failures are treated as the implementation backlog, not as test noise. The failing groups listed above are reproduced before product changes begin.
2 Enforce declaration rules before field resolution. Named declarations are the only names reachable through ${params.name}. Invalid declaration names fail validation. Positional parameters do not become namespaced params. ValidateUnsupportedLocations/invalid_declaration_name and ValidateUnsupportedLocations/positional_params_are_not_addressable_by_name pass.
3 Validate supported params references everywhere Spec 003 allows value resolution. Declared ${params.name} references are accepted. Undeclared ${params.missing} references fail validation with field context. Unbraced $params.name text is preserved as ordinary content. ValidateDeclaredReferences remains green, and ValidateRejectsUndeclaredReferencesByField plus ValidatePreservesUnbracedNamespaceTextByField reach zero failures.
4 Reject params references where params are unavailable. Literal declaration fields and root consts list values do not gain access to runtime params. Non-Dagu braced text remains ordinary string content. ValidateUnsupportedLocations/params_defaults_do_not_support_references passes, and preservation fixtures remain green.
5 Resolve declared runtime params at field-use time. Every supported root, step, handler, environment, precondition, stream, output, and action-input string covered by Spec 005 sees the supplied runtime param value before the field is used. Start/named_runtime_params_resolve_in_string_fields and StartResolvesFieldSurfaces reach zero failures.
6 Fail missing runtime values before the owning field is used. A declared params reference without a runtime value produces a non-zero run result and prevents the field's side effect from happening. StartRejectsMissingRuntimeValues reaches zero failures.
7 Apply Spec 005 rules inside the params[].eval boundary. params[].eval may use declared ${params.name} references according to the dynamic-evaluation specs, and undeclared or missing references fail before the evaluated parameter is consumed. ValidateParamsEvalReferences and the params_eval runtime cases pass.
8 Protect compatibility behavior. Existing bare-name and environment expansion behavior remains available where specs permit it, but it does not satisfy or hide ${params.name} references. Existing non-Spec-005 tests remain green, and conformance failures do not move into compatibility fixtures.
9 Close the status loop. Spec 005 can be marked implemented only after validation, runtime, missing-value, and compatibility boundaries all pass through the conformance entry point. make test-conformance and the normal non-conformance test target both pass.

Conformance Verification Plan

  • Use make test-conformance as the only required entry point for binary-level Spec 005 verification.
  • Keep the current exhaustive field matrix intact: declared-name validation, undeclared-name validation, unbraced-text preservation, successful runtime resolution, and missing-runtime-value failure.
  • Keep accepted runtime coverage exceptions explicit and justified; do not use them to hide fields that are easy to verify through local side effects.
  • Assert failures by behavior class and field context. Do not couple conformance to private error formatting beyond the context needed to identify the invalid field.
  • Verify the normal test target still excludes binary-level conformance tests so missing DAGU_BIN cannot create silent skips or false green runs.
  • After each workstream, run the narrow failing group first, then run the full conformance target before claiming progress.

Accepted Runtime Exceptions

Validation conformance still covers every Spec 003 field surface. Runtime conformance covers every field with a stable local side effect, except for the explicitly accepted cases below.

Exception Accepted boundary
Root, step, and handler container fields Successful and missing-value runtime checks would cross into Docker or existing-container state before producing a stable local side effect.
Step and handler parallel variable, items, and item params Successful and missing-value runtime checks require sub-DAG fan-out orchestration; validation still covers declared and undeclared authoring, and unbraced namespace-looking text is preserved as ordinary content.
Step and handler repeat-policy conditions Successful and missing-value runtime checks are coupled to repeat-count behavior after command execution; validation remains exhaustive.
Step and handler stdout or stderr artifact paths Artifact paths resolve inside run-managed artifact storage whose concrete path includes run identity; stdout and stderr local path cases cover stream redirection.
Step and handler stdout output field paths The current YAML grammar does not accept file path sources under stdout outputs; structured step output paths cover file path output resolution.
Handler stdout output values and handler output entries Handler output values are terminal handler metadata with no downstream step that can consume them as a stable local file side effect.
handler_on.abort and handler_on.wait runtime execution Abort requires external cancellation and wait requires an approval transition; validation still crosses handler steps with the step-owned field matrix.

Risk Register

Risk Mitigation
Current unresolved ${params.*} behavior is accidentally preserved as the target. Keep Spec 005 as the source of truth and require tests that prove declared names resolve.
Bare-name compatibility changes the new namespace semantics. Test bare names and ${params.name} separately, especially omitted values and positional parameters.
Validation and runtime support different field lists. Use exhaustive field-list conformance tests: validation covers every Spec 003 field surface, and runtime covers every runtime-verifiable field except the accepted runtime exceptions listed above.
Missing runtime values fail too late. Acceptance tests must prove the owning field is not used after a missing ${params.name} value is detected.
Foreign-language variable syntax is mistaken for Dagu syntax. Keep Dagu-owned value resolution limited to braced supported forms. Treat unbraced namespace-looking text such as $params.name as ordinary content, with only a non-fatal warning if a warning channel is added later.
Spec status is overstated. Keep Spec 005 marked not implemented or partial until every required behavior has conformance coverage.
